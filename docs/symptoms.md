# 典型的な状態

状態は感情名よりも挙動で記述したほうが扱いやすくなります。  
ここでは、外から見える変化と内側で起きやすい処理遅延を分けて整理します。

## 表面に出やすい変化

### 応答が遅くなる

返事をしないのではなく、返事を生成する工程が増えています。  
短文ですら `入力 -> 解釈 -> 危険判定 -> 返答候補の選別` を通るため、平常時より時間がかかります。

### 語調が急に平板になる

怒っているように見えても、実際には余白を消しているだけの場合があります。  
装飾語が減り、句読点が増え、説明より断定が残ります。

### 予定の変更に極端に弱くなる

内容の難しさより、切り替え回数の多さで消耗します。  
「軽いお願い」が連続すると、単発の大きな負荷より早く飽和します。

## 内側で起きやすい処理

### 予測失敗の連鎖

会話や予定の先読みが外れると、次の処理に余計な計算が入ります。  
その結果、目の前の話題より「次に何が来るか」の監視が優先されます。

### 自責の自動実行

失敗の事実より先に、原因の引き受けが走ることがあります。  
これは反省というより既定動作に近く、停止には外部の明示が必要です。

## 記録例

短いメモは、感情語より条件を残すほうが後で読み返せます。

`会話がつらい` より `二人以上の会話で、話題転換が三回を超えると追従できない` のほうが有効です。

```txt
時刻: 21:40
入力: 雑談の途中で予定確認が入った
反応: 返答停止、視線固定、あとから自己嫌悪
回復条件: 10分の無言、選択肢を2つまでに減らす
```

## 観測時の注意

- 単発の言葉より、直前30分の流れを見る
- 沈黙を拒絶と解釈しない
- 結論を急ぐ前に、入力過多の可能性を確認する
